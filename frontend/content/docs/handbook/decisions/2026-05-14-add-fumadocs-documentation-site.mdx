# 2026-05-14 — Add Fumadocs documentation site at `/docs`

<Callout type="info" title="Status: Proposed">
Add a Fumadocs-powered documentation surface to the existing `frontend/` Next.js app, serving a public-facing internal **handbook** at `/docs/handbook/**` and a user-facing **product** section at `/docs/product/**`. Single repo, single Next app, single deployment.
</Callout>

---

## Context

Today, `docs/` at the repo root is a mixed-purpose dumping ground:

- Durable, public-safe reference: `decisions/` (ADRs), `agents/` (agent
  guidance), `ci/` (self-hosted runner notes), `deployment/`.
- Working notes: `plans/`, `solutions/`, `superpowers/`, `hyperframes/`,
  plus loose top-level `.md` files. These often reference unshipped work
  and debug findings.
- Binary assets: `assets/`.

There is no rendered documentation site. ADRs and agent guidance only
surface via raw markdown links in source control. New users of pawrrtal
(the chat product) have nowhere to read about models, modes, or
settings.

This decision introduces a documentation site that:

1. Renders the durable, public-safe subset of repo-root `docs/` as a
   browsable handbook.
2. Ships a small seed of product documentation aimed at end users.
3. Lives inside the existing `frontend/` Next app so there is no second
   deployment to maintain.

## Goals

- A public, indexable docs surface at `/docs` with two sections
  (`/handbook`, `/product`).
- Curated handbook content drawn from the durable subset of repo-root
  `docs/`, moved into the Next app under `frontend/content/docs/`.
- Five product pages seeded in the same PR.
- Stock Fumadocs theme — no design-system integration in v1.
- No new deployment target, no new auth surface, no new secrets.
- Local search via Fumadocs's built-in Orama index. No third-party
  search.

## Non-goals

- Full product documentation. We seed five pages; expansion is
  follow-up work.
- Visual parity with the pawrrtal app shell. Choice taken on
  2026-05-14: stock Fumadocs theme. See
  [Alternatives considered](#alternatives-considered).
- Translations / i18n.
- Versioned docs (per-release branches).
- Migrating `plans/`, `solutions/`, `superpowers/`, or any other
  working-note folder. These stay at repo root as the raw scratchpad
  that agents read and write.
- Touching `DESIGN.md`, `CLAUDE.md`, or `AGENTS.md` content. These stay
  at the repo root for their respective consumers (the design linter,
  Claude Code, OpenAI Codex). The handbook may add thin pages that
  link to them.

## Architecture

```
                 ┌──────────────────────────────────┐
                 │   frontend/ (Next 16 App Router) │
   public        │                                  │
   visitors  ──► │  /  /(app)  /(auth)  /settings   │  ← pawrrtal app (unchanged)
                 │  /docs                           │  ← new Fumadocs surface
                 │       /docs/handbook/[[...slug]] │
                 │       /docs/product/[[...slug]]  │
                 │       /api/search                │  ← Orama (built-in, local)
                 │                                  │
                 │  fumadocs-mdx (build-time plugin)│
                 └──────────────────────────────────┘
                              │
                              ▼ reads at build & request time
            ┌──────────────────────────────────────────────┐
            │  frontend/content/docs/                      │
            │     handbook/   ← `git mv` from docs/        │
            │     product/    ← 5 new MDX files            │
            └──────────────────────────────────────────────┘
```

**Boundaries:**

- Fumadocs touches only `frontend/`. No backend or FastAPI changes.
- `RootProvider` (from `fumadocs-ui/provider/next`) wraps the app at
  the root layout. The existing `<Providers>` (React Query, etc.) nests
  *inside* `<RootProvider>`.
- `RootProvider` uses `next-themes` internally. We configure it to read
  the existing `dark` class on `<html>` so the existing FOUC-safe
  blocking script (`public/theme-detection.js`) remains the source of
  truth. No two competing theme systems.
- Search is local (Orama, bundled with `fumadocs-core/search/server`).
  No new env vars, no Algolia.

**Stack compatibility (verified 2026-05-14):**

- `fumadocs-ui@16.8.10` peer-deps `next: "16.x.x"`, `react: "^19.2.0"`.
- `fumadocs-mdx@15.0.4` peer-deps `next: "^15.3.0 || ^16.0.0"`,
  `react: "^19.2.0"`.
- Pawrrtal: Next 16.2.6, React 19.2.6. Clean fit.

## Routing model

Two loaders, two route segments, one shared layout:

```ts title="frontend/lib/source.ts"
import { handbookDocs, productDocs } from 'collections/server';
import { loader } from 'fumadocs-core/source';

export const handbookSource = loader({
  baseUrl: '/docs/handbook',
  source: handbookDocs.toFumadocsSource(),
});

export const productSource = loader({
  baseUrl: '/docs/product',
  source: productDocs.toFumadocsSource(),
});
```

`app/docs/layout.tsx` resolves which `source` to use from the active
pathname and passes `source.getPageTree()` to `<DocsLayout>`. The two
`[[...slug]]/page.tsx` files render their respective MDX bodies via
`source.getPage(params.slug)`.

## Provider composition & theme integration

```tsx title="frontend/app/layout.tsx"
<html lang="en" suppressHydrationWarning className={...fontVars}>
  <head>
    <Script src="/theme-detection.js" strategy="beforeInteractive" />
    {/* unchanged */}
  </head>
  <body>
    <RootProvider
      theme={{
        attribute: 'class',
        defaultTheme: 'system',
        enableSystem: true,
        // FOUC script already set the class — next-themes reads it
      }}
    >
      <Providers>{children}</Providers>
      {process.env.NODE_ENV === 'development' && <Agentation />}
    </RootProvider>
  </body>
</html>
```

The existing blocking theme script remains the FOUC defence.
`next-themes` (inside `RootProvider`) initialises *from* the class the
script set, so a cold load on either theme produces a single,
consistent paint.

## Product seed content (5 pages)

Feature-tour style. Each ~200–400 words, no marketing fluff.

1. **`index.mdx` — Introduction.** What pawrrtal is, who it's for, one
   sentence on the model-agnostic approach.
2. **`conversations.mdx` — Conversations & Sidebar.** Creating new
   conversations, sidebar navigation, search.
3. **`models.mdx` — Models.** Which providers are supported, how to
   pick a model, where the model selector lives.
4. **`modes.mdx` — Modes & Reasoning.** Plan mode, safety mode,
   reasoning levels: what each does, when to use it.
5. **`settings.mdx` — Settings.** Tour of the settings surface
   (appearance, account, etc.), pointing back to product features
   referenced elsewhere.

## Public access, SEO, search

- **Auth:** `/docs/**` is fully public. No middleware change needed
  (pawrrtal does not gate routes via Next middleware today; backend
  cookie auth governs API access only).
- **SEO:** `public/robots.txt` allows `/docs/**` and disallows `/api/`.
  `app/sitemap.ts` enumerates the union of
  `handbookSource.getPages()` and `productSource.getPages()` plus the
  `/docs` landing.
- **Search:** `app/api/search/route.ts` exposes
  `createFromSource(handbookSource, productSource)` — Orama index built
  at build time, served from the same Next deployment, no external
  service.

## Testing

- **Build smoke:** `bun run build` must succeed. This catches MDX
  frontmatter issues, broken `createRelativeLink` calls, and missing
  pages.
- **Route smoke** (single Node script or Stagehand spec): `/docs`,
  `/docs/handbook`, `/docs/product`, `/docs/handbook/decisions/&lt;known-slug&gt;`,
  `/api/search?query=models` all return HTTP 200 and contain the
  expected `&lt;title&gt;` element.
- **Source-tree unit test** (`frontend/test/source.test.ts`):
  `handbookSource.getPages().length` equals the count of MDX files
  under `content/docs/handbook/**`. Same check for product. Catches
  drift where someone adds an MDX file but the loader doesn't pick it
  up (frontmatter missing, wrong location, etc.).
- **No new Playwright/Stagehand e2e** in this PR. The dev-console
  smoke (`scripts/dev-console-smoke.mjs`) already gates console
  errors; `/docs/handbook` and `/docs/product` are added to its
  cold-boot route list.
- **`just sentrux`** must stay green. New files live in established
  sentrux layers (`app/`, `lib/`, `components/`); no new edges.
- **`bun run design:lint`** is unaffected (DESIGN.md stays put).

## Phasing (single PR, five commits)

1. `chore(docs): scaffold fumadocs deps + config` — install packages,
   `source.config.ts`, `next.config.mjs` wrap, `tsconfig` alias. No
   routes yet.
2. `feat(docs): scaffold /docs routes and providers` — `app/docs/**`
   route tree, `lib/source.ts`, `components/mdx.tsx`, `RootProvider`
   wrap, CSS imports, robots/sitemap. Empty
   `content/docs/handbook/index.mdx` + `product/index.mdx`.
3. `feat(docs): migrate handbook content` — `git mv` + frontmatter +
   path sweep.
4. `feat(docs): seed product section (5 pages)` — five MDX files.
5. `feat(docs): search route + sitemap` — `/api/search` and full
   `sitemap.ts`.

Each commit independently green (`bun run check` + `bun run build`).

## Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| `RootProvider` next-themes fights existing FOUC `dark`-class script | medium | Configure `RootProvider` with `attribute="class"` so next-themes reads the existing class. Smoke-test cold-load on both themes. |
| `fumadocs-mdx` (ESM-only) forces `next.config.ts` → `.mjs` migration that breaks other config | low | Inspect `next.config.*` before converting; preserve all existing options in the new file. |
| File-line budget (500 LOC) for `app/docs/[[...slug]]/page.tsx` | very low | Boilerplate is ~40 LOC. |
| Sentrux flags `lib/source.ts` importing from `collections/*` | low | `collections/*` aliases `.source/*`, a generated build artefact already gitignored. If sentrux flags it, add to its ignore list. |
| Pre-existing `app/sitemap.ts` collision | unknown | Verify before writing; extend if present. |
| Path-sweep misses a reference | medium | Post-sweep `grep -rE 'docs/(decisions\|agents\|ci\|deployment)/' .` and inspect every hit. Whitelist intentional historical mentions. |
| Public surface exposes content that shouldn't be public | low after curation | Manual eyeball pass over each moved file before merge. The moved folders are already public-safe by content type (ADRs, agent guidance, deployment notes). |

## Alternatives considered

- **Standalone docs project.** Spin up a separate Next app under
  `apps/docs/` or as a sibling repo. Rejected: extra deployment, extra
  CI, no benefit for this scale.
- **Render everything in repo-root `docs/` verbatim.** Rejected:
  exposes unshipped plans and debug writeups on a public surface.
- **Keep `docs/` at repo root as source of truth; include selected
  subfolders.** Rejected (Question 3, option B) because per-folder
  include config is fiddly and the "wrong file in the wrong folder"
  failure mode silently leaks content.
- **Copy instead of move.** Rejected (Question 6, option B) — drift
  between two copies of every ADR is a known anti-pattern.
- **Move with redirect stubs at old paths.** Rejected (Question 6,
  option C) — stub clutter doesn't pay for itself given the one-time
  path-sweep is mechanical.
- **Login-gated docs.** Rejected (Question 5, option B) — public docs
  for a public product is the standard model, and curation already
  strips sensitive content.
- **Full design-system integration.** Rejected (Question 4, option C)
  — fighting Fumadocs's CSS surface on every minor upgrade is a known
  trap; visual distinction is appropriate for a docs surface.

## Open questions

- `/docs` landing: a two-card chooser (handbook vs product) or a
  redirect? **Default:** two-card chooser, ~30 LOC, friendlier than a
  redirect.
- Whether to surface `CLAUDE.md` and `AGENTS.md` in the handbook as
  thin link pages, alongside the `DESIGN.md` link page. **Default:**
  not in v1; revisit if there's demand.

Neither is load-bearing; both can be settled during implementation.

## Follow-ups (not in this PR)

- Reconsider design-system integration if visual distinction becomes a
